# テストのために仕様を歪めない

## 適用条件

テストコードを新規作成・変更するとき。特に**フレーキー（不安定テスト）・実行速度・テスト環境の制約に対処するとき**に厳守する。

## 背景

移植元プロジェクトでの事例（PRD-278: テスト検証基盤の信頼性回復、2026-07-13）で、`updated_at > created_at`（更新は必ず作成より後・二回更新すれば後の更新日時は確実に未来 = プロダクトの真の仕様）を検証する assertion を、テスト環境（WSL2）の壁時計 backward step で稀に fail する flaky を避けるため `updated_at !== created_at`（値が異なるだけ）へ弱めようとした。これは `updated_at < created_at` という実バグを見逃す**劣化テスト**であり、テスト環境の事情を仕様の側へ押し付ける**仕様の歪曲**だった。flaky の原因はテスト環境（時刻源の非単調性）にあり、直すべきは原因の側であって、仕様を表す assertion ではない。

## 制約

1. **テストは仕様を忠実に表明する**: assertion が表す不変条件は、プロダクトの仕様が要求する強さを保つ。順序性・厳密等価・必須性などを、テストの都合で弱い述語へ置き換えない（例:「更新後は作成より後」は `>`。`!==` へ弱めない）。
2. **フレーキー・遅さ・環境差異は原因を直す**: これらへの対処は、仕様を表す assertion ではなく、その原因（テスト環境・ハーネス・時刻源・並行性・isolation・mock 設計など）を修正して行う。
3. **即時に根本対応できない場合の暫定手続き**: (i) 仕様を歪めない範囲の暫定回避に留め、(ii) 根本対応を `TODO` へ起票し、(iii) 暫定であることをテスト付近のコメントに明示する。仕様を弱めた assertion を「対処済み」として放置しない。

## 禁止事項

- テスト環境の都合（フレーキー回避・実行速度・環境依存）を理由に、assertion が表す**仕様の意味論を弱める・すり替える**こと（順序性 `>` を同値否定 `!==` に、厳密等価を近似許容に、必須を任意に、網羅を代表値に、等）。
- 仕様が要求する性質を検証しない形へテストを書き換えて「green だから対処完了」とすること（劣化テストの見逃し）。
- フレーキーの**原因**（環境・ハーネス・時刻源）を直さず、**症状**（assertion）だけを緩めて隠すこと。

## 検証

- 変更後の assertion が、変更前と**同等以上の強さ**で仕様を表明している（弱められていない）。
- フレーキー対処が「原因（環境・ハーネス）の修正」になっており、仕様側を緩めていない。緩めざるを得なかった場合、暫定である旨のコメントと根本対応の `TODO` が存在する。
