# 設計品質ゲート

## 適用条件

設計判断を伴う場面で常時適用される判断軸を定める。具体的には:

- PRD 起草前の既存割れ窓調査（`prd-template` skill §3.2 から参照）
- リファクタリング判断・割れ窓検出（`refactoring-check` skill から参照）
- コード変更箇所の周辺点検全般

本 rule は **path-scope rule**（コード作業時に auto-load: `apps/*/src/**` 等の src を Read/Edit すると発火）。設計判断はコード作業時に効くため src を file トリガにして適時ロードし、計画・会話など非コード作業時には載せない。加えて `prd-template` §3.2 / `refactoring-check` 等から link 参照される。

## 制約

### 基本原則（CLAUDE.md Core Principles）

- **KISS**: 過剰設計を避ける。最小限の複雑さで現在の要件を満たす
- **YAGNI**: 要求されていない機能・改善・拡張を作らない。今必要なものだけを実装する
- **DRY + 直交性**: DRY は「知識の重複」を排除する原則であり、「コードの見た目の重複」を排除する原則ではない。共通化によってモジュール間の結合が増えるなら、重複を残す
- **対称性**: 抽象レベルで同一視すべきものは同一の構造・扱いで表現する。扱いを変えるなら抽象レベルの実際の差異を反映していなければならない（恣意的な非対称も偽の対称も排除）

### 既存割れ窓の調査観点

修正・開発箇所の周辺コード（同一ファイル / 同一ディレクトリ / 同一ドメインの api / renderer / store / view）を grep / Read で点検し、以下のような既存の割れ窓を抽出する:

**A. 実装欠如・誤り系**
- エラーハンドリング欠如（API 呼び出しの try-catch なし、console 突き抜け）
- frontend / backend バリデーション乖離（片側のみ実装）
- 古い API 形式の残骸 / dead code
- 認可漏れ
- 同種 UI 改修箇所で他の field がリンク化されていない等の非対称

**B. 設計品質系（DRY / 対称性 / 直交性 / 結合度）**
- **DRY 違反 (知識の重複)**: 同じビジネスロジック / 計算式 / 定数 / バリデーションルール（の意味）が複数箇所に重複している。**判定基準は CLAUDE.md「DRY は知識の重複を排除する原則であり、コードの見た目の重複を排除する原則ではない。共通化によってモジュール間の結合が増えるなら、重複を残せ」**。例:
  - 同じドメインルール（バリデーション規則・状態遷移の条件等）を複数 handler が独自実装
  - frontend / backend で計算式・変換ロジックが独立して書かれており、片方の変更がもう片方に追従しない
  - ⚠️ frontend regex と backend regex のような「shape の重複」は **defense-in-depth として意図的なら DRY 違反ではない**（frontend の入力型と backend の Zod スキーマの二重管理は、境界ごとの防御として意図的でありうる）
- **対称性違反**: 抽象的に同種のものが不揃いに扱われている、または別物が無理に同型化されている。DRY「見た目の重複を共通化するな」の双対で、「同種なら扱いを揃えよ／非対称には理由を持て」。例:
  - 並列な兄弟（リソース別 handler / enum・union の各ケース / 同種 UI フィールド群）の一つだけが別の形・別の扱いになっている（A 観点の「同種 UI 改修箇所で他 field 未リンク」もこの構造的非対称）
  - 逆操作の対が欠けている / 非整合（create はあるが delete が別扱い、serialize と deserialize の round-trip 不整合、subscribe/unsubscribe の対応漏れ、renderer の initialize と dispose の非対称）
  - union/enum の一部ケースだけを特別扱いし、その非対称が意図的だと明示されていない
  - ⚠️ 非対称を破ること自体は禁止ではない。破るなら「なぜ違う扱いが必要か」を言語化すること。また YAGNI と衝突する「対称だから未使用の逆操作も今作る」先取りはしない
- **直交性違反**: 本来独立すべき関心事が同じ関数 / モジュール / コンポーネントに混在している。例:
  - ビジネスロジックと presentation logic が view に同居
  - Babylon の描画状態管理と Vue/Pinia の reactive state が混在し、境界（`BabylonCanvas`）が曖昧になる
  - 1 つの service / renderer が複数の責務を持つ（God object）
- **過剰結合**: 1 つの変更が連鎖して多数のモジュールを修正する構造になっている。例:
  - shared mutable state にコンポーネント群が依存し、改修時に全件追従が必要
  - 共通 helper が呼び出し元の都合に応じて分岐し続けている（snowball helper）
  - 修正時に「ファイル grep して全部置換」で済むのは見た目の重複であって、知識の DRY とは別問題

### 共通化候補を棄却する前の高次元抽象探索（義務）

共通化候補を「見た目の重複 / 偽の対称」と棄却する前に、抽象度を 1 段上げて**差分を分類**する:

- **(a) 射影の差**: 戻り値の形・codomain だけが違う（列を返すか件数を返すか等）
- **(b) 評価時点の差**: 即時評価か遅延評価か（式エンジン・コールバックへの供給等）
- **(c) 文脈の差**: 入力・依存する文脈が呼び出し元ごとに違う
- **(d) 規則・不変条件そのもの**: ドメイン規則・計算式・不変条件が同一か

**(d) が共有されているなら知識の重複**であり、(a)〜(c) だけが異なる場合は「コア（知識）+ アダプタ（差分）」で知識部分のみ統一できないかを検討してから判定する。逆に (d) が異なるなら、(a)〜(c) が揃って見えても共通化しない（偽の対称）。この探索は双方向に効く — 「見た目が違うから別物」も「見た目が同じだから統一」も、(d) の同一性を確認するまで確定しない。

実例（本プロジェクトの構造）: WebGL / WebGPU の 2 backend は生成 API・フォールバック挙動が表層で異なるが、「Engine を生成して描画ループへ渡す」という知識は 1 つであり、`createEngine` の単一 seam に (a) 射影・(c) 文脈の差として吸収され、呼び出し側（`BabylonRenderer`）は backend 差を意識しない（コア + アダプタで知識を統一した例）。いっぽう Vue/Pinia の reactive state と Babylon の毎フレーム描画状態は、同じ「状態」に見えても更新頻度・所有権という (d) の本質差があり、統一せず `shallowRef` で境界を保つ（偽の対称を作らない例）。

### 抽出結果の振り分け

抽出した割れ窓は以下のいずれかに振り分ける:

- **本作業で同時修正**: 「本来のスコープを大きく逸脱しない」範囲のものはスコープに含める（PRD の場合は「対象」セクションに**割れ窓修正項目**として明記する）
- **逸脱するため別 PRD / TODO**: 大きすぎるものは `TODO` に詳細記録（修正先パス / 規模 / 本作業との関連）

### フロントエンド web プラットフォーム準拠（modern-web-guidance）

本 rule の path-scope は backend も含むが、**この節はフロントエンドの UI / レイアウト / CSS / クライアント JS 作業（`apps/frontend/src/**`・`.vue`・css・DOM/Web API を扱うコード）にのみ適用する**。

web プラットフォーム（HTML/CSS/Web API）は陳腐化が速く、学習済みの古いパターンを温存しやすい。該当作業では `modern-web-guidance` スキルを設計品質ゲートの一観点として用いる:

- **実装着手前**: 実装する前に `/modern-web-guidance` を先行実行し、対象テーマ（モーダル/ポップオーバー、スクロール連動アニメ、`:has()`/コンテナクエリ、フォーム、Core Web Vitals 等）の現行ベストプラクティスを取得する。手順・対象範囲・retrieve 機構の正はスキル本体（`.agents/skills/modern-web-guidance/SKILL.md`）に委ねる。
- **レビュー時**: 既存/変更コードが旧パターン（JS 手管理で代替できる新 CSS 機能の不使用など）を採っていないかを、本ゲートの A/B 観点に加えて点検する。

この観点は実装フローでは `/tdd`、レビューフローでは `/review-dev` から運用される（各 skill 本体を正とする）。

## 禁止事項

- **「コードの見た目が同じ」だけを根拠に DRY 違反と判定して共通化を強行すること**（CLAUDE.md「DRY は知識の重複を排除する原則」に反する。共通化で結合が増えるなら重複を残す）
- **高次元抽象探索（差分の (a)〜(d) 分類）を行わずに共通化候補を「見た目の重複 / 偽の対称」と棄却すること**（棄却は (d) 規則・不変条件の不一致を確認してから。表層の差〔射影・評価時点・文脈〕だけを根拠に「別物」と断定しない）
- 抽象的に同種のものを恣意的に不揃いに扱う／別物を無理に同型化することを、理由の言語化なしに行うこと（対称性違反）
- 周辺コードの割れ窓調査をスキップすること（実装欠如系だけでなく **DRY / 対称性 / 直交性 / 結合度の設計品質系** も対象）
- 将来の拡張を見越した設計を先取りで盛り込むこと（YAGNI）
- 抽出した割れ窓を「本作業で修正もしない・TODO にも記録しない」まま放置すること（第 3 の出口禁止）

## 検証

- 設計判断の対象箇所について、A（実装欠如・誤り系）/ B（設計品質系）両観点の点検結果が存在する
- 抽出した割れ窓が「本作業で修正」か「`TODO` / 別 PRD へ記録」のいずれかに振り分けられている
- DRY 違反の判定が「知識の重複」基準でなされ、「見た目の重複」を根拠にした共通化強行がない
- 共通化候補を棄却した判定に、差分分類（射影 / 評価時点 / 文脈 / 規則）による根拠が示されている（規則の不一致を確認せず棄却していない）
- 並列な兄弟・逆操作の対・union 全ケースについて、非対称があれば「意図的で理由が言語化されている」か点検されている（対称性）
