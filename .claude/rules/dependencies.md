# 依存関係のバージョン管理

## 適用条件

`package.json` に依存を追加する、または既存の依存を更新するとき。

## 制約

1. 追加・更新する依存の最新バージョンを確認する
2. 最新バージョンを指定する
3. 既存の依存との整合性を確認する（`npm install` + `npm run type-check` が通ること）
4. 整合性が取れない場合、関連する依存も含めて最新に揃える
5. バージョン固定よりも最新追従を優先する

バージョン方針の正は README.md（Node 24 系固定・PostgreSQL 18 系・npm latest 追従）。

## 挙動を変えうる差し替え・版上げの安全網（characterization 先行＝拡張 TDD）

ライブラリの**差し替え（別ライブラリへの置換）**や、**挙動・UI に影響しうる版上げ**は、見た目の回帰・微妙な挙動差が起きやすい。ライブラリ差し替え・挙動影響のある upgrade は、対象モジュールの vitest 特性テスト（現挙動の固定）を先に書いてから行う（`/tdd` の拡張適用）:

1. 差し替え対象が現に提供している**ユーザー向け挙動（操作・表示・発火イベント・戻り値）**を、切り替え**前に** vitest の characterization テストで固定して GREEN 化する。
2. ライブラリを差し替え、同テスト群が **GREEN のまま**であることで挙動等価を確認する。
3. 新規挙動を伴う差分は通常の RED→GREEN→REFACTOR に従う。

> characterization は**故意のミューテーションで RED になる（非空虚）**ことを確認し、安全網が実際に回帰を検出できる状態にしてから切り替える。

## 禁止事項

- 理由なく古いバージョンを指定すること
- 最新バージョンの確認をせずに依存を追加すること
- **挙動・UI に影響しうる差し替え・版上げを、現行挙動の characterization 安全網を用意せずに行うこと**（切り替え後の等価性を GREEN で担保できない）
