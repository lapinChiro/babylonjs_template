# PRD テンプレート

## トリガー

`backlog/` に新しい PRD を作成するとき。

## アクション

### 1. TODO のバッチ化検討

PRD 化対象を決めた直後、Discovery を始める前に必ず実施する:

1. `TODO` を全件スキャンし、対象と**影響範囲が近い・被る項目**を抽出する。判定軸:
   - **共通ファイル / 共通モジュール**を編集する（同じ component / 同じ schema / 同じ migration）
   - **共通の概念モデル**に触れる（同じテーブル / 同じ画面 / 同じドメイン）
   - **依存関係**がある（A の実装後に B が成立する、A と B がコンフリクトしうる）
2. 候補がある場合、ユーザーに「A と B を 1 PRD にまとめるか / 分けるか」を提示する。判断軸:
   - **まとめる利点**: 共通コードの重複編集回避、テストの一括化、レビュー効率
   - **分ける利点**: 独立したリリース / ロールバック単位、レビュー粒度の管理
3. ユーザー確認後、バッチ化対象を確定してから Discovery に進む

### 2. Discovery

PRD を書き始める前に、以下を行う:

1. 最低 2 つの明確化質問をユーザーに投げる。**PRD の規模に応じて追加する**:
   - なぜ今これを作るのか（動機・優先度の確認）
   - 何をもって成功とするか（完了条件の認識合わせ）
   - 制約はあるか（技術的制約、既存機能との整合性、スケジュール等）
   - **DB migration / global UI 影響 / 既存 PRD への依存等を伴う場合は、ロールバック方針・段階的展開・影響範囲の事前確認も Discovery に含める**
2. 回答を得てから PRD を起草する

### 3. 設計品質チェック (起草前)

PRD を起草する前に、以下を機械的に確認する。スキップ禁止。

#### 3.1 「本来あるべき実装」の確認

「本来あるべき実装」とは、**frontend と backend の適切な場所でそれぞれ適切な処理が実装され、エラーが必ずハンドリングされている**実装を指す。本プロジェクト（Vue 3 + Babylon.js + Hono + PostgreSQL + MinIO のテンプレート）の構成に照らして確認する:

- **バリデーション**: frontend は vee-validate/yup で即時 UX feedback のため、backend は @hono/zod-openapi の Zod スキーマで security / 整合性の最終ガードのため、**両層に実装**する
- **エラーハンドリング**: API 呼び出し / 非同期処理は必ず catch し、ApiError への変換を経て**ユーザー可視 UI** に伝える（`console.error` 突き抜け禁止）
- **認証**: backend は JWT middleware で必ず実施。frontend UI は**表示制御のみ**で security は持たせない
- **データ整合性**: DB 制約（NOT NULL / UNIQUE / FK）で型・値域強制を担う。application layer は前提として書く

PRD の「設計」セクションが上記いずれかを欠いている場合、起草前に補完する。

#### 3.2 既存割れ窓の調査

修正・開発箇所の周辺コード（同一ファイル / 同一ディレクトリ / 同一ドメインの api / repository / view）を grep / Read で点検し、既存の割れ窓を抽出する。**点検観点（A. 実装欠如・誤り系 / B. 設計品質系＝DRY・直交性・結合度）と DRY 判定基準（知識の重複か、見た目の重複か）の正は [`.claude/rules/design-quality-gates.md`](../../rules/design-quality-gates.md) を参照**。

抽出結果を以下のいずれかに振り分ける:

- **本 PRD で同時修正**: 「本来のスコープを大きく逸脱しない」範囲のものはスコープに含める。PRD の「対象」セクションに**割れ窓修正項目**として明記する
- **逸脱するため別 PRD / TODO**: 大きすぎるものは `TODO` に詳細記録（修正先パス / 規模 / 本 PRD との関連）

#### 3.3 起草前のセルフチェック

起草前に PRD 案について以下を確認:

- [ ] frontend / backend 両側のバリデーション実装が含まれている（片側のみで済まないか再考）
- [ ] エラーハンドリング (catch + UI 表示) が含まれている
- [ ] 周辺の既存割れ窓のうち、修正対象に含めたものを「対象」セクションに明記している
- [ ] 修正対象外とした既存割れ窓は `TODO` に追記したことを「対象外」セクションに記載している

### 4. PRD 起草

起草前に [`referent-before-label`](../../rules/referent-before-label.md) を Read し、対応表（referent table）を scratchpad へ独立保存してから本文を書く（新規ファイルの Write では path-scope が発火しないため能動 Read が必要）。

以下のテンプレートに従って記述する:

```markdown
# <タイトル>

## 背景・動機

なぜこの機能が必要か。現状の課題や、この機能がないことで生じている問題。

## ゴール

この PRD が完了したとき、何ができるようになっているか。具体的・検証可能な形で書く。
曖昧な表現（「速い」「簡単」「直感的」等）を避け、具体的な数値・閾値・観測可能な挙動で記述する。

## スコープ

### 対象

この PRD で実装する範囲を箇条書きで列挙する。

### 対象外

明示的に対象外とするものを列挙する。スコープクリープを防ぐ。

## 設計

### 技術的アプローチ

実装方針。既存のアーキテクチャとの関係、変更が必要なモジュール、新規追加するモジュール。

### 影響範囲

変更が及ぶファイル・モジュールの一覧。

### ロールバック / スキーマ整合

DB migration を含む PRD で記述する（含まない場合は各項目に「該当なし」と明示する。読者が「考慮し忘れ」と「該当なし」を区別できるようにする）:

- **対象データへの影響**: 既存データ（migration 内で seed するデモデータを含む）の変更有無を明記。なければ「影響なし」と書く
- **ロールバック手順**: 失敗時の復旧手順。DB migration は up/down 対で書き、戻し方（`npm run migrate:down` による reverse migration）を具体的に記述する
- **スキーマ整合**: DB スキーマの変更内容と、**保存データが不正化しないことの根拠**（DB 制約 = NOT NULL / UNIQUE / FK・適用前の状態確認・必要な migration 適用順序）を書く

## 作業ステップ

TDD で進める前提で、RED → GREEN → REFACTOR の順序を意識したステップに分解する。
各ステップは独立してテスト可能な単位にする。

- [ ] ステップ1: ...
- [ ] ステップ2: ...

## テスト計画

追加・変更するテストの概要。正常系・異常系・境界値を含む。

## 完了条件

この PRD の作業が「完了」と言える条件。品質チェック（lint / type-check / knip / test / build）を含む。

## 完了後処理

本 PRD の完了処理は、`/tdd` ステップ0 で登録する**完了処理タスク列**に従って実行する（列の登録・順序の正は `/tdd` ステップ0）。以下は列の**最終タスク `/prd-completion`** が行う PRD 固有のファイル処理チェックリスト（本 PRD 固有の instance）:

- [ ] `backlog/{file}.md` を削除
- [ ] `plan.md` の消化順序・進行中から該当エントリを削除
- [ ] `plan.md` 末尾「完了 PRD」index の最上段に「| {N} | {短タグ} |」を追記（短タグに実装詳細を列挙しない。index の形式・保持件数・最古行削除の不変条件は `backlog-layer-model` を正とする）
- [ ] 関連する `TODO` 項目が削除済み（or `TODO.archive` 退避済み）であることを再確認
- [ ] **PRD 番号参照 cleanup** (`.claude/rules/no-prd-references-in-code.md` 準拠): 以下の grep を実行し、本 PRD で新規追加・修正したファイルおよびその周辺で `PRD-{N}` の文字列が残っていないことを確認する。hit があれば設計意図のみの記述に書換 or 削除する:
  ```bash
  grep -rEin "PRD-?{N}\b" apps/*/src apps/*/tests apps/*/scripts
  # 例外パス (規約で許可): backlog/, plan.md, .claude/rules/*.md, report/, TODO
  ```
```

## 禁止事項

- Discovery（明確化質問）をスキップして PRD を書くこと
- **TODO のバッチ化検討をスキップして PRD を書くこと**（影響範囲が被る項目を見逃すと、後続 PRD で同じファイル・同じ migration を二重に編集する手戻りが発生する）
- **設計品質チェック（3.1 / 3.2 / 3.3）をスキップして PRD を起草すること**
- **対応表（referent table）を独立保存せずに PRD 本文を起草すること**（[`referent-before-label`](../../rules/referent-before-label.md)。書き始めてしまったら本文を破棄し、表の保存からやり直す）
- **アドホックなその場しのぎ実装を PRD で許容すること**（「とりあえず動く」「ローカルで if 文を増やす」等は割れ窓の起点。本来の層で対応する設計に書き直す）
- **frontend バリデーションだけで完結させ backend を欠落させること** / **その逆**（両層必須）
- **API エラーをコンソールに rethrow するだけで UI に反映しないこと**（ユーザー可視のエラー表示を必ず含める）
- **修正・開発箇所の周辺コードに対する割れ窓調査をスキップすること**（同一ファイル / 同一ディレクトリ / 同一ドメインを点検する。点検観点＝実装欠如系 / 設計品質系の詳細は [`design-quality-gates`](../../rules/design-quality-gates.md)）
- **「コードの見た目が同じ」だけを根拠に DRY 違反として共通化を強行すること**（判定基準の正は [`design-quality-gates`](../../rules/design-quality-gates.md) / CLAUDE.md Core Principles）
- 曖昧な完了条件（「適切に動作する」「問題なく使える」等）を書くこと
- 将来の拡張を見越した設計を PRD に盛り込むこと（YAGNI）
- 1 つの PRD に**独立した**機能を複数詰め込むこと（影響範囲が共通でバッチ化合意済みの機能群は 1 PRD に統合してよい — 「独立」は影響範囲が被らないことを指す）
