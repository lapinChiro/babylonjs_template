# TDD 開発手順

## トリガー

新機能の追加またはバグ修正に着手するとき。

## アクション

> **フロント先行調査**: 着手対象がフロントエンドの UI / レイアウト / CSS / クライアント JS(`apps/frontend/src/**`・`.vue`・css・DOM/Web API を扱うコード)を含む場合、**実装着手前に `/modern-web-guidance` を先行実行**し、現行ベストプラクティスを取得してからテスト設計・実装に進む(web プラットフォームは陳腐化が速く古いパターンを温存しやすい。観点の正は `.claude/rules/design-quality-gates.md` フロント節)。
>
> **UX 先行設計**: 着手対象が user-facing な UI を伴う場合、**着手時に `/ux-design` discovery を実施**し、Tier 判定 + ジョブストーリー(Tier に応じてインパクトマップ / フロー分析)で「何を作るか」を固めてからテスト設計に進む(PRD ありの場合は `/prd-template` §3.4 で実施済のため重複しない)。

以下の順序で実装する:

0. **タスク登録**(PRD 着手時 / 複数ステップ作業時は必須): 着手前に、以下 (i)(ii) の 2 群をタスク管理ツール(`TaskCreate`/`TaskUpdate`、または `TodoWrite`)に登録する。以降、各ステップ開始時に `in_progress`、検証込みで完了した時点で `completed` に更新し、着手後に発生した追加作業(指摘修正・ファイル分割・再検証)もその都度タスク化する。制約の正は `.claude/rules/task-tracking.md`。
   - **(i) 実装ステップ**: PRD 本文の「作業ステップ」(PRD 以外は計画したステップ)。
   - **(ii) 完了処理タスク列**(実装ステップの後に続けて実行する**固定列**。本列の順序の正はここ。PRD・複数ステップ作業・単発修正のいずれも同経路):
     1. `/refactoring-check`
     2. `/review-dev`
     3. `/ux-design` review(**UI を伴う変更のみ**。UI 変更なし=T0 ならスキップ。ヒューリスティック評価 + 認知的ウォークスルー + a11y。指摘の修正は本ステップで行い後段の `/quality-check` で検証する)
     4. `/quality-check`(**フルモード**=完了報告を支える検証)
     5. `/todo-audit`
     6. `/check-local`(実機環境がなく**実行不能**なら、勝手にスキップ・完了扱いにせず**ユーザーにヒヤリング**してから進める)
     7. `/prd-completion`(軽量=PRD/TODO/plan.md ファイル処理 + PRD 番号 cleanup + 完了報告。**必ず最後**。前段 1〜6 の全検証が通過してからでないと plan.md を完了確定しない)
1. **テスト設計**: `/test-design` skill を呼び出し、本ステップを完全に終えるまで RED に進まない
   - `/test-design` のステップ 1〜8 をすべて実行する
   - 境界値分析 / 状態遷移 / デシジョンテーブル / ペアワイズ / エラー推測を MECE に適用する(純粋ロジックへの property-based 併用は fast-check 導入後に適用。正は `/test-design` §2.4)
   - テスト項目テーブル (具体的入力値 + 期待結果 + 適用技法) を **コード実装前に** 作成する
   - 本ステップを「正常系・異常系・境界値を 2〜3 件思いついた」レベルで済ますことは禁止 (`/test-design` の禁止事項を参照)
2. **RED**: 設計したテスト項目テーブルをテストコードに落とし、失敗することを確認する
   - `cd apps/<app> && npx vitest run <test_file> -t "<test_name>"` で対象テストだけ実行する(両 app とも vitest)
   - テスト名は `describe('<対象>') > it('<条件>のとき<期待結果>')` 構造を守る
   - **バグ修正の場合**: 最初の RED は報告された実障害をそのまま再現する失敗テストにする。修正コードを書く前に、そのテストが **実障害ゆえに fail する** ことを確認する。再現できない場合は原因が未特定とみなし、修正に進まない
3. **GREEN**: テストを通す最小限のコードを書く
4. **REFACTOR**: テストが通る状態を維持しながらリファクタする
   - ステップ完了(タスクを `completed` にする)前に `/quality-check` の**軽量モード**で検証する(変更ファイルの lint / 変更 app の type-check / 影響テスト / 依存更新時のみ knip)。軽量モードは fast-feedback で完了根拠にはしない(完了処理列の `/quality-check` フルモードが正)

## 禁止事項

- 実装コードを先に書き、テストを後から書くこと
- `/test-design` を呼ばずに、または途中で省略してテスト設計を済ますこと
- テスト項目テーブルを作らずにテストコードを書き始めること
- 境界値・状態遷移・リスク領域の網羅を「主要なものだけ」で省略すること
- テストを書いた後に RED(失敗)を確認せず GREEN に進むこと。テストが最初から通る場合、そのテストは何も検証していない可能性がある
- 期待結果を事前に定義せず、実行結果を見てから「これが正しい」と後付けで判断すること
- バグ修正で、実障害を再現する回帰テストを書かずに(または fail 再現を確認せずに)修正コードを書くこと

## 検証

- `/test-design` skill のテスト項目テーブルが PRD / テストコメント / TODO のいずれかに永続化されている
- 実装コードの差分より前にテストコードの差分が存在する(git history で確認可能)
- テスト名が `describe('<対象>') > it('<条件>のとき<期待結果>')` の構造で検証内容を表現している
- `/test-design` の検証セクションを満たしている (境界値 5 種類 / 状態遷移表 / リスク領域チェック)
- バグ修正の場合、実障害を再現する回帰テストが存在し、修正前は fail・修正後は pass することを確認している
