# ルールの書き方

## トリガー

ルールの**書き方（構造・記法）を確認・適用するとき**。rule の作成・更新・削除という**操作の入口は `/rule-maintenance`** で、本 skill はその中で参照される「書き方リファレンス」（個別作成時の単独利用も可）。役割分担: `rule-maintenance` = メンテ手順（個別レビュー + 全体統合チェック）/ `rule-writing` = 1 ルールの構造・記法ガイド。

## ルールの2類型

ルールは以下の2種類に分類される。種類に応じて必須セクションが異なる。

| 種類 | 性質 | 例 |
|------|------|-----|
| **手続きルール** | 特定のイベントで発動し、具体的なステップを実行する | 「作業完了時に品質チェックを実行する」 |
| **制約ルール** | 常時適用される条件・禁止事項を定める | 「検証時には期待結果を事前に定義せよ」 |

手続きルールは **スキル** として配置する（`.claude/skills/<name>/SKILL.md`）。
制約ルールは **ルール** として配置する（`.claude/rules/<name>.md`）。

## ロード機構（いつコンテキストに読み込まれるか）

配置とあわせて「いつ読み込まれるか」を設計する。機構は以下で確定している（実測）:

| 配置 / frontmatter | 読み込まれるタイミング |
|---|---|
| **skill**（`.claude/skills/`） | `/name` 起動時のみ（`description` を見て呼ぶか判断）。ファイル編集では自動ロードされない |
| **rule・frontmatter なし** | **常時ロード**（毎セッション開始時・全タスク） |
| **rule・`paths:` あり** | 該当 glob のファイルを **Read / Edit した時**に自動ロード（適時） |

**確定している性質（実測）:**

- **Skill ツールでの skill 起動は、その SKILL.md を「Read」扱いにしない。** よって rule の `paths` に skill ファイルを指定しても、その skill を**起動しただけでは発火しない**（skill ファイルを明示的に Read した時のみ）。「skill 実行時にこの rule を載せたい」を `paths` では実現できない。
- **ルールのレジストリはセッション開始時に構築される。** rule の新規作成・frontmatter 変更は**次セッションから**有効（当セッションには反映されない）。
- **リンク掲示（本文に `[X](X.md)` と書くだけ）では自動ロードされない。** ロードには上表のいずれかの経路（常時 / `paths` 一致 / 明示 Read）が要る。

**制約ルールの「常時 / 適時」の判断軸:**

- **(a) どのタスクでも必要、かつ (b) 専用 skill/rule をロードする"前"に必要** → frontmatter なし（常時ロード）。例: 検証原則・タスク管理・横断 gotcha・どの skill を呼ぶかのルーター。
- **特定の作業ファイルを触る時だけ必要** → `paths` で適時化（例: TZ 規約は DB/repo 編集時、テスト規約は test 編集時）。
- **特定 skill 実行時だけ必要だが file トリガが無い** → その skill 本文が「`X.md` を Read せよ」と**能動 Read を指示**する（`paths` では起動に紐づけられないため）。
- 常時ロードは毎セッション全タスクで課金される。**task-specific 知識を no-frontmatter で積まない**（[`documentation-drift-prevention`](../../rules/documentation-drift-prevention.md) §5: 常時ロード文書は orientation のみ、手順・列挙はポインタへ委譲）。

## 構造

### 手続きルール（スキルとして配置）

| セクション | 必須 | 内容 |
|------------|------|------|
| **トリガー** | 必須 | いつ発動するか。具体的なイベントや状態を記述する |
| **アクション** | 必須 | 何をするか。機械的に実行できる具体的ステップ |
| **禁止事項** | 推奨 | やってはいけないこと。抜け道になりがちな行動を列挙する |
| **検証** | 推奨 | ルールに従ったかの判定基準。第三者が客観的に判定できる形 |

手続きルールでトリガーが書けない場合、そのルールは抽象的すぎるか、制約ルールとして書き直すべきである。

### 制約ルール（ルールとして配置）

| セクション | 必須 | 内容 |
|------------|------|------|
| **適用条件** | 必須 | どのような場面で常時適用されるか |
| **制約** | 必須 | 守るべき条件・原則。具体的な行動指針として書く |
| **禁止事項** | 推奨 | 制約に反する典型的な行動パターンを列挙する |

## 原則

- **LLM は「やるべきこと」だけ書くと都合よく解釈する**。禁止事項を書かないと、ルールの文面に反しない別の方法で意図を逸脱する。「削除する」と書いても「打ち消し線で残す」を許容してしまう。禁止事項でこれを防げ
- **曖昧な指示は無視される**。「適切に対応する」「注意する」「考慮する」は実質的に何も指示していない。具体的な行動に書き換えよ
- **1 ルール 1 関心事**。複数の関心事を 1 ファイルに混ぜない。関心事が増えたら分割する

## 関連

操作の入口 `rule-maintenance` との役割分担はトリガー節参照。書く際にあわせて従う rule:

- [`rule-retroactive-application`](../../rules/rule-retroactive-application.md): rule/skill を**新規作成**するときは、本体と同時に既存コードへの遡及適用（+ 分割時は `TODO` 起票）を行う。
- [`documentation-drift-prevention`](../../rules/documentation-drift-prevention.md): 常時ロード文書の bloat 回避・source/skill への委譲（ロード機構の「常時/適時」判断と対）。
