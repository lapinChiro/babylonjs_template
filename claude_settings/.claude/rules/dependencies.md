# 依存関係のバージョン管理

## 適用条件

`package.json` に依存を追加する、または既存の依存を更新するとき。

## 制約

1. 追加・更新する依存の最新バージョンを確認する
2. 最新バージョンを指定する
3. 既存の依存との整合性を確認する（`npm install` + `npx tsc --noEmit` が通ること）
4. 整合性が取れない場合、関連する依存も含めて最新に揃える
5. バージョン固定よりも最新追従を優先する

## 挙動を変えうる差し替え・版上げの安全網（characterization 先行＝拡張 TDD）

ライブラリの**差し替え（別ライブラリへの置換）**や、**挙動・UI に影響しうる版上げ**は、見た目の回帰・微妙な挙動差が起きやすい。これらは「切り替え前に現行挙動を固定する安全網を先に用意し、切り替え後も GREEN を保つことで等価性を担保する」手順を踏む（TDD を切り替えに拡張。実施は `/tdd`→`/test-design` 経由）:

1. 差し替え対象が現に提供している**ユーザー向け挙動（操作・表示・発火イベント）**を、切り替え**前に** characterization テストで固定して GREEN 化する。
   - **UI ライブラリ**の場合、Storybook Story が安全網を兼ねる（本プロジェクトは `@storybook/addon-vitest` の `storybookTest` で Story が headless Chromium のブラウザテストになる＝`vitest run --project storybook`）。視覚・操作・発火イベントを Story / interaction test で固定する。
   - **非 UI（ロジック寄り）**の場合は通常の unit characterization で固定する。
2. ライブラリを差し替え、同テスト群が **GREEN のまま**であることで挙動等価を確認する。
3. 新規 UI / 挙動を伴う差分は通常の RED→GREEN→REFACTOR に従う。
4. **視覚回帰のみ**（出力が等価で内部経路だけ変わる類）の差し替えは、Story のビジュアル確認 + 既存 component test の GREEN 維持で代替してよい。

> characterization は**故意のミューテーションで RED になる（非空虚）**ことを確認し、安全網が実際に回帰を検出できる状態にしてから切り替える。

## 禁止事項

- 理由なく古いバージョンを指定すること
- 最新バージョンの確認をせずに依存を追加すること
- **挙動・UI に影響しうる差し替え・版上げを、現行挙動の characterization 安全網を用意せずに行うこと**（切り替え後の等価性を GREEN で担保できない）
