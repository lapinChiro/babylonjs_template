# 複数ステップ作業のタスク管理ツール追跡

## 適用条件

PRD（`backlog/` の PRD）へ着手するとき、または 3 ステップ以上を要する開発作業（機能追加・リファクタ・複数ファイル横断修正・レビュー指摘の一括対応など）に着手するとき。

## 背景

タスクを頭の中だけで管理すると、後から発生したステップ（指摘修正・検証の再実行）が抜け落ちる。2026-06-14、PRD-122 で「API テスト編集後に backend lint を再実行する」を明示タスク化していなかったため、早い段階の lint 結果（編集前・上限内）を最終状態と誤認し、`max-lines` 超過を見逃して CI で初めて検出された。ステップを可視タスクとして登録し状態を更新していれば、「未完了の再 lint」が残ることに気づけた。

## 制約

複数ステップ作業では、タスク管理ツール（`TaskCreate` / `TaskUpdate` / `TaskList`、または `TodoWrite`。1 作業内ではいずれか一方で一貫させる）で全ステップを追跡する:

1. **着手時に全ステップを登録する**: 着手前に **(i) 実装ステップ**（PRD は本文の「作業ステップ」、そうでなければ計画したステップ）と **(ii) 完了処理タスク列**（refactoring-check 〜 prd-completion の固定列。列の正は `/tdd` ステップ0）の両方をタスクとして登録する。完了処理列の最後の `/prd-completion`（軽量）は前段の全検証が通過してから実行する。
2. **状態をリアルタイムに更新する**: 各ステップ開始時に `in_progress`、完了時に `completed` へ更新する（`in_progress` は同時に 1 つだけ）。事後にまとめて状態を辻褄合わせしない。
3. **「完了」は検証込みで判定する**: ステップを `completed` にできるのは、そのステップに対応する検証（quality-check の lint / type-check / knip / test、動作確認等）まで通った時点。実装しただけ・検証前は `completed` にしない（`.claude/rules/verification.md`）。
4. **後から発生した作業もタスク化する**: 着手後に判明した追加ステップ（レビュー指摘、未計画の修正、ファイル分割等）は、その時点でタスクに追加してから着手する。
5. **検証通過後の再編集には「再検証」タスクを立てる**: 検証通過後にファイルを再編集したら、影響 workspace の quality-check 再実行を新規タスクとして登録する（`.claude/rules/local-quality-mirrors-ci.md`: 検証は最終状態で行う）。

## 禁止事項

- タスクを登録せずに複数ステップ作業へ着手すること。
- ステップの実態と異なる状態を放置すること（着手済みなのに `pending` / 検証前なのに `completed`）。
- 検証前のステップを `completed` にすること（実装完了 ≠ ステップ完了）。
- 着手後に発生した指摘・追加修正を、タスクに反映せず記憶・口頭だけで進めること。
- 検証通過後にファイルを再編集したのに「再検証」タスクを立てず、古い検証結果を最終結果として完了報告すること（本ルール制定の契機となった事象）。
- セッション終盤にまとめてタスクを作ること（事後の辻褄合わせ。リアルタイム更新の趣旨に反する）。

## 検証

- 作業開始直後のタスクリストに、PRD / 計画の全ステップが登録されている。
- 各ステップの状態遷移（`pending` → `in_progress` → `completed`）が作業実態と時系列で一致している。
- 完了報告時点で、未検証・再検証待ちのステップが `completed` になっていない。
- 着手後に発生した指摘・追加修正が、タスクとして追記されている。
