# 過渡期のスキーマ不整合は許容し、保存データの正しさを守る

## 適用条件

wire 契約（API schema / frontend の API 型）または DB スキーマを変更するとき。および PRD の作成・レビューでそれらの変更を設計・評価するとき。

## 背景

frontend / backend / batch / DB は独立したデプロイ単位のため、リリースの過渡期にはスキーマの不整合が必ず生じうる。この不整合を「起きてはならないもの」として扱うと、旧形式の二重受理・旧クライアント向け fallback・段階的移行フラグ・deploy 順序への設計依存が積み上がり、**一時的な事象のために恒久的な複雑さ**を抱え込む。

2026-07-15、ユーザーが判断基準を明示した: **過渡期の不整合は許容する（そのために実装へ複雑さを導入しない）。ただし DB に保存されたデータの正しさは保証されなければならない**。前者は一時的で deploy 完了により解消するが、後者は不可逆だからである。この非対称が本 rule の全体を貫く。

## 制約

### 1. 過渡期の不整合は許容する

deploy の過渡期に生じる frontend↔backend / backend↔DB のスキーマ不整合（＝一時的なエラー・機能の一時不全）は許容する。これを回避するための実装を導入しない:

- 旧形式と新形式の二重受理
- 旧クライアント向けの fallback 分岐・互換シム
- 段階的移行のためのフラグ・feature toggle
- 「片側先行でも壊れない」ことを保つための設計上の拘束

### 2. DB に保存されたデータの正しさは保証する

過渡期であっても、**不正なデータが永続化してはならない / 保存済みデータが壊れてはならない**。スキーマ変更が保存データを不正にしうるなら、防御を設計に含める:

- 不変条件は **DB 制約（unique / CHECK / FK）で backstop する**。app 層の事前 check だけに委ねない（concurrent race が check をすり抜ける）
- 既存データが新しい制約を満たすことを **適用前の監査で確認する**（満たさなければ流さずユーザーへ確認する）
- **deploy 順序は「保存データの正しさに必要な場合のみ」制約として課す**（可用性・互換のためには課さない）

### 3. 境界の判定

| 過渡期に起きること | 扱い |
|---|---|
| エラーが返る / 機能が一時的に使えない / 画面が壊れる | **許容**（回復可能・deploy 完了で解消する） |
| 不正なデータが保存される / 保存済みデータが壊れる | **不許容**（不可逆・設計で防ぐ） |

### 4. PRD での書き分け

- **「後方互換」を受入条件・テストの pin にしない**。副産物として壊れないことは事実として記録してよいが、保証すると将来の変更を縛る制約になる
- **「保存データの正しさ」は受入条件に据える**（何がそれを守るか＝ DB 制約・適用前監査・テストを示す）

## 禁止事項

- 過渡期の不整合を吸収するための互換コード（旧形式の二重受理・旧クライアント向け fallback・移行フラグ）を新規に導入すること
- 「旧クライアント / 旧 deploy との互換」を根拠に optional・既定値・分岐を新設すること（**省略時既定値そのものは禁止でない。根拠が「過渡期の互換」であることが禁止**）
- 「後方互換」「deploy 順序制約なし」を PRD の受入条件・テストの pin にすること
- 保存データの不変条件を app 層の事前 check だけに委ね、DB 制約 / backstop の要否を評価しないこと
- スキーマ変更時に「過渡期に不正データが保存されうるか」を評価せずに完了とすること
- 可用性・互換の維持を理由に deploy 順序を制約として課すこと（順序制約は保存データの正しさに必要な場合に限る）

## 検証

- PRD の受入条件に「後方互換の維持」が含まれていない（含まれるのは「保存データの正しさ」）
- 新設した optional / fallback / 分岐の根拠が「過渡期の互換」でない
- スキーマ変更に対し、過渡期に保存データが不正化しないことの根拠（DB 制約・適用前監査・deploy 順序の必要性）が示されている
- deploy 順序を制約として課している場合、その理由が保存データの正しさである
