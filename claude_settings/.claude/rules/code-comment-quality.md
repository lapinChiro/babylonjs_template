# コードコメントの品質

## 適用条件

ソースコード・テスト・スクリプト・インフラ定義ファイル（本 rule の `paths`）でコメント / docstring を追加・変更するとき、および既存コメントをレビューするとき。

## 制約

コメントは「コードから読み取れないこと」だけを書く。書いてよい内容は次の 3 種:

1. **制約・不変条件**: コード自身が示せない前提（「この順序でないと X が壊れる」「分母は正が保証済み」等）
2. **設計意図（なぜ）**: なぜこの実装・この形を選んだか（採らなかった代替案と理由を含む）
3. **外部要因**: 外部 API の仕様・既知バグ・プラットフォームの落とし穴など、コードの外にある知識

コメントを書く・変更する・レビューするときの品質観点:

- **管理用番号を含めない**: PRD 番号・タスク番号等の管理用番号をコメント・テスト名・識別子に残さない（PRD 番号の具体的な判断基準・grep 手順・例外パスの正は [`no-prd-references-in-code`](no-prd-references-in-code.md)。本 rule はレビュー観点として指す）
- **実装と同期させる**: コメントが指す関数・値・挙動が現存するかをコード変更と同じ編集で確認し、実装を変えたらその実装を説明する周辺コメントも同時に追従させる（stale コメントは誤情報として実害を生む）
- **自明の言い換えを書かない**: 直後のコードを自然言語に訳しただけの説明・関数名から自明な説明は書かない（見つけたら削除する）

## 禁止事項

- 「何をするか」をコードの言い換えとして書くこと（コードを読めばわかる）
- 変更の経緯・レビュー向け説明（「〜を修正」「〜から移設」「この変更が正しい理由」等、マージ後にノイズになる記述）を書くこと。経緯の正は git history / commit message
- 実装変更時に、その実装を指す既存コメントの追従確認を省略すること
- stale と気づいたコメントを「後で直す」と放置すること（即修正 or 即削除。修正がコメントの域を超える場合のみ `TODO` 起票）
- 管理用番号（`PRD-N` 等）をコメント・テスト名・識別子に書くこと（禁止の詳細は `no-prd-references-in-code` が正）

## 検証

- 追加・変更したコメントが「制約・不変条件 / 設計意図 / 外部要因」のいずれかを述べている（コードの言い換え・経緯説明ではない）
- 変更した実装を指す周辺コメントが、変更後の実装と一致している
- 管理用番号が `no-prd-references-in-code` の検証 grep で検出されない
