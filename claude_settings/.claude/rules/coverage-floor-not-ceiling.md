# 列挙した観点・項目は最低限の floor（十分条件ではない）

## 適用条件

レビュー・検証・テスト設計などで「観点」「確認項目」「チェックリスト」を**列挙して網羅性を担保する**場面で常時適用される判断軸を定める。具体的には:

- `review-dev` / `review-prd` の必須観点
- `check-local` の確認項目（正常系 / 異常系 / 境界値）
- `verification` の検証項目洗い出し

本 rule は **path-scope rule**（レビュー/検証の作業ドメインを Read/Edit すると auto-load: コード `**/src/**` / PRD `backlog/**` / テスト `tests/**`）。加えて `review-dev` / `review-prd` / `check-local` / `verification` から link 参照される。`verification`（常時ロード）は file トリガを持たない検証にも適用されるため、その本文ポインタ経由で本 rule を能動 Read して補う。

## 制約

- スキル等に**固定列挙された観点・項目は「どの対象でも必ず見る最低限（floor）」**であり、それだけで十分（網羅）とはみなさない。floor は最低限であって上限ではない。
- 各対象について、まず**対象そのものを読み**、その対象固有のリスクから**追加の観点・項目を動的に導出**して floor に足し込む。導出の手がかり（対象により該当するものを選ぶ）:
  - 扱うドメイン / データ構造 / 不変条件
  - 外部連携・I/O（API・DB・ファイル・メッセージング）
  - 権限・認可・セキュリティ境界
  - 移行・後方互換・データパッチ
  - 並行性・順序・冪等性
  - エラー処理・失敗時の挙動・部分失敗
  - パフォーマンス・スケール
  - 運用・観測性・ロールバック
- **floor + 動的に導出した追加観点の合算**を、対象に MECE に割り当ててから着手する。
- 追加観点は「対象のどのリスクに対応するか」を根拠として示せる形にする（恣意的な思いつきの羅列ではなく、対象から導出されたもの）。

## 禁止事項

- 固定列挙（floor）だけをチェックして「全観点を見た／網羅した」とすること。
- 対象を読まずに（固有リスクを把握せずに）レビュー・検証計画を確定すること。
- 「列挙（floor）に無いから見ない」と、列挙の不在を未確認の正当化に使うこと。
- 追加観点の導出を「該当なし」で済ませた際に、その判断根拠を示さないこと。

## 検証

- レビュー・検証計画に、floor（固定列挙）に加えて**対象固有の追加観点**が含まれている（または「追加なし」の根拠が示されている）。
- 各追加観点について、対象のどのリスクから導出したかが示されている。
