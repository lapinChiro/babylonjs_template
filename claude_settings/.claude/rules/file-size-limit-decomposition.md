# ファイル/関数の行数上限超過は本質的分割で応える

## 適用条件

lint の `max-lines`（ファイル行数）/ `max-lines-per-function`（関数行数）の上限を超過したとき、または編集により超過が予見されるとき。frontend (`*.vue` / `*.ts`) / backend (`*.ts`) いずれも対象。

## 背景

行数上限は「そのファイル/関数が過大な責務を抱えている」という設計シグナルである。PRD-96 Step 7 で `ReviewDetailView.vue` が上限を超えた際、当初「名前付き computed (`isReviewSubmitted`) を template 式へ inline 化する」「コメントを削る」等の**行数を削るためだけのアドホックな修正**で上限を回避しようとした。これは可読性・テスタビリティを下げる割れ窓であり、上限が本来示していた「責務過大」という問題を隠蔽するだけだった。

正しい対応は、Overview セクション（プロジェクト/スケジュール表示 + 折りたたみパネル state）を `ReviewOverviewSection.vue` に**本質的に分割**することだった。これはレビュー編集/提出ロジックと直交し自己完結するため、抽出により結合度低下・直交性向上・テスタビリティ向上が得られ、副作用として行数も大きく下がった（545→414 行）。

## 制約

行数上限を超過したら、超過分を削る前に必ず以下を行う:

1. **俯瞰分析**: 当該ファイルと関連する周辺ファイルを読み、`<section>` / `<!-- ... -->` / 機能ブロック / 責務の境界を洗い出す。
2. **凝集単位の特定**: 結合度・直交性・DRY・テスタビリティ・純粋関数（stateless/stateful 分離）の観点で、**自己完結し他と直交する責務の塊**を見つける。
3. **本質的分割**: その塊を子コンポーネント（`*.vue`）/ 別モジュール（`*.ts` の factory・純粋関数）へ抽出する。抽出先には対応する単体テストを追加する。
4. **スコープ判断**: 当該作業単位で適切な分割が収まらない場合は、リポジトリ root の `TODO` に**具体的な対象と分割方針を起票**する（「後で直す」等の曖昧な記述は不可。`.claude/rules/rule-retroactive-application.md` の分割適用と同基準）。

行数を下げることは目的ではなく、責務分割の**結果**として行数が下がる。「上限に収めること」を目的化してはならない。

## 禁止事項

- **名前付き computed / 変数 / 関数を template 式やインラインへ畳んで行数を稼ぐ**こと（可読性・テスタビリティの後退。例: `const isReviewSubmitted = computed(...)` を `:prop="review.x !== null && ..."` に inline 化）。
- **意味のあるコメントを削って行数を稼ぐ**こと。
- **複数行を無理に 1 行へ詰めて**行数を稼ぐこと。
- 上限超過を**眼前の関心事でないからと放置**すること（既存実装の割れ窓も対象。本質対応するか TODO 起票する）。
- 分割の検討をせずに「とりあえず収まったから良い」と判断すること。

## 検証

- 行数上限超過を解消した変更で、解消手段が「凝集単位の抽出（新規コンポーネント/モジュール/純粋関数 + そのテスト）」または「TODO 起票」のいずれかである。
- 解消手段に「named binding の inline 化」「コメント削除のみ」「1 行詰め込みのみ」が含まれない。
- 抽出した単位に対応する単体テストが追加されている。
