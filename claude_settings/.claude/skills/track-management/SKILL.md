# トラック管理

## トリガー

トラック（共通ゴールへ向けて複数 PRD を連続消化する前向き計画単位）を**新規作成・保守・中断・終了**するとき。単発 PRD（どのトラックにも属さない 1 項目）の作成は本 skill の対象外で、`prd-template` / `backlog-replenishment` を直接使う。

## 構造モデル（正は rule）

トラックの**定義**・**JIT PRD 不変条件**・**トラックは任意（開発の 3 モード）**・layer-model との関係の**正は [`backlog-layer-model`](../../rules/backlog-layer-model.md)** を参照する。本 skill はそのモデルを操作する**具体手続き**を定める。`plan.md` / `backlog/` / `TODO` への書き込み機構（層間整合・完了 index・命名規約等）は [`backlog-management`](../backlog-management/SKILL.md) skill へ委譲する（本 skill はトラック構成の判断に専念する）。

## アクション

### 0. トラック化すべきか判断（任意）

トラックは必須でない。以下を**両方**満たすときにトラック化し、満たさなければ単発 PRD で足りる:

- 共通ゴールへ向けて **2 つ以上の PRD 単位**が見込まれる
- 単位間に**順序依存・レバレッジ・伝播リスク**があり、消化順序を明示する価値がある

満たさない孤立課題は `TODO` から単発 PRD 化する（トラックを作らない）。

### 1. 必須要素を定める

トラック作成時、以下を `plan.md` の前向きセクションに明記する。小トラックを重くしないため「必須トリオ + 条件付き」の二層とする。

**必須（常に）:**

1. **ゴール**: 完了時に何ができるようになるか（検証可能・観測可能な形。曖昧表現を避ける）。トラックの真の完了条件（DoD）が「全単位消化」と異なる場合はここに明記する
2. **スコープ境界（非ゴール）**: トラックが**やらない**ことを明示し、トラック単位のスコープクリープを防ぐ（例: 「ガントは対象外」）
3. **単位分解**: 構成する PRD 単位への分解（垂直分割 backend/frontend・段階 Step1/2/3 等）。**単位名 + 一言**で置く（PRD ファイルは作らない＝JIT）。分解は**暫定であり、先行 PRD の学びで改訂しうる**ことを前提とする
4. **消化順序 + 根拠**: 単位の順序と選定根拠（§2 の決定手続き）

**条件付き（該当時のみ）:**

5. **設計メモ / 不変条件**: 横断で効く設計知識（集計代数・母集団意味論・データ可用性制約等）を `report/` に置き「正」とする
6. **前提・要検証事項**: スキーマ未強制の仮定・実測前提など

### 2. 消化順序を決める

単位の消化順序は、以下の**優先順位付き手続き**で決める（上から順に適用し、上位の判断が下位を上書きする）。総合的な重み付けでなく precedence にするのは、判断のブレを抑えトラック品質を安定させるため（[`todo-prioritization`](../../rules/todo-prioritization.md) の 3 軸を、トラック内の順序決定用に優先順位付けした形）。

1. **ハード依存（トポロジカル制約）**〔絶対〕: 単位 B が A なしに着手不能なら B は A の後（例: frontend は backend API の後）。最適化でなく制約なので最上位。
2. **レバレッジ（基盤先行）**: 後続単位の工数を減らす基盤（共有抽象・指標抽象・wrapper 等）を先に。
3. **伝播防止**: 放置すると後続で workaround 負債が N 倍化する単位を先に。
4. **律速・スケール・リスク規模**: データ可用性が浅い次元は後ろへ／小さく速い単位を先に／大規模・高リスクな実装は基盤が揃う後半へ。ただし**前提そのものが未検証（approach が成立するか不明）**な単位は、早期の検証単位（spike）で不確実性を潰すことを検討する。
5. **直接価値**〔tie-breaker〕: 上記で決まらない範囲を価値順で。

補足ルール:

- **(D) 再評価（JIT との整合）**: 上記で決めた順序はトラック作成時点の暫定。**次単位を材料化（PRD 化）する直前に本手続きで再評価**する（先行 PRD の学びを反映。[`analysis.md`](../../rules/analysis.md) の螺旋・再評価と整合）。
- **(E) 希望順の上書き**: 本手続きは**推奨順 + 根拠**を出力する。product owner（ユーザー）は上書きでき、**上書き時は根拠を `plan.md` に記録**する（分析順を機械的に強制しない）。

### 3. plan.md へ記述

- トラックを `plan.md` の前向きセクションに追加する（plan.md 更新手続きは `backlog-management` に従う）
- 単位分解は**単位名 + 一言**で置く。PRD ファイルは作らない（JIT）
- 横断の設計メモがあれば `report/` に置き、`plan.md` からはリンクで指す

### 4. 材料化（次単位を PRD 化）

- 着手直前に、次に消化する **1 単位のみ**を `prd-template` で PRD 化し `backlog/` へ置く（＋進行中があればそれも）
- 前倒しで後続単位を PRD 化しない（JIT PRD 不変条件・正は rule）

### 5. 中断・保留

トラックを途中で止めるとき（方針転換で別トラックへ移る等）:

- 残単位の扱いをユーザーと決める: (a) `TODO` へ課題として畳み戻す、(b) `plan.md` の保留セクションに残単位を残す
- `backlog/` に材料化済みで**未着手**の PRD があれば、JIT 原則に沿って破棄し、`TODO` / `plan.md` の単位記述へ畳み戻す（作り置きを残さない）

### 6. 終了

- **終了条件はゴール／DoD の達成**。全単位の消化はその典型ケースに過ぎない（単位分解は暫定＝§1-③のため、ゴール達成が全単位消化と乖離しうる）。
- DoD 達成時に**未消化の単位が残っていれば、§5（中断・保留）に準じて畳み戻す**（`TODO` へ戻す／材料化済み未着手 PRD は破棄）。
- トラックを `plan.md` 前向きセクションから除去する（完了痕跡は「完了 PRD」index の 番号 + 短タグ のみ・`backlog-management` に従う）

## 禁止事項

- 未着手の後続単位を前倒しで `backlog/` に PRD ファイル化して作り置きすること（JIT PRD 不変条件違反）
- 孤立した単発課題を無理にトラック化すること（トラックは任意・2 単位未満はトラック化しない）
- 消化順序の**根拠を書かず**に順序だけ並べること
- トラック中断時に、材料化済み PRD や残単位を宙ぶらりんのまま放置すること（`TODO` / `plan.md` へ畳み戻す）
- `plan.md` の前向きセクションに完了痕跡・後ろ向き prose を残すこと（`backlog-layer-model` の前向き不変条件に従う）

## 検証

- トラックが `plan.md` 前向きセクションにあり、単位分解が**単位名 + 一言**で置かれている（未着手単位が PRD ファイル化されていない）
- `backlog/` に「次の 1 本（＋進行中）」を超える未着手 PRD の作り置きが無い
- 消化順序に**根拠**（3 軸 + 対象固有の律速）が併記されている
- 中断したトラックの残単位が `TODO` か `plan.md` 保留セクションで追跡できる
