# テスト設計手順

## トリガー

以下のいずれかに該当するとき。**テストコードを書く前に必ず本手順を完了**する:

- テストファイルを新規作成する前
- 既存テストファイルに新規テストケースを追加する前
- バグ修正のための回帰テストを追加する前
- TDD のステップ 1 (テスト設計) の中

## アクション

以下の順序で **すべて** 行う。ステップを飛ばすことは禁止。ただし、ステップ 1 で該当しないと判定した特性のステップは「該当なし」と明記してスキップ可。

> 前提: 仕様が不明確な場合は、不明点を明示し、仮定を記載した上で設計を進める。仮定はテスト項目テーブルの備考列に記録する。

### 1. テスト対象の特性分類

テスト対象が以下のどれに該当するか確定する (複数該当あり):

| 特性 | 該当例 | 主に適用する技法 |
|---|---|---|
| **入力 → 出力** | 純粋関数 / SUM 集計 / バリデーション / フォーマッタ | 同値分割 + 境界値分析 (BVA)。純粋ロジックは property-based 併用 (§2.4) |
| **状態 + イベント → 状態** | state machine / workflow / レビュー進行 / 認可遷移 | 状態遷移テスト (STT) |
| **複数条件分岐** | 認可判定 / 業務ルール / フォールバック分岐 | デシジョンテーブル (DT) |
| **複数パラメータ組合せ** | API 引数組合せ / 設定オプション / フォーム入力 | ペアワイズ |
| **並行 / 競合** | DB 競合 / TX race / cache invalidation | エラー推測 + シナリオテスト |
| **時刻 / 環境依存** | 期間集計 / Date 計算 / i18n / タイムゾーン | エラー推測 + リスク領域追加 |

### 2. 同値分割と境界値分析 (BVA)

入力パラメータごとに以下を行う:

#### 2.1 同値分割クラスの列挙

入力ドメインを「同じ振る舞いを期待する集合」(= 同値クラス) に分割する。例: 整数の estimate 値なら `{負値}` `{0}` `{正の値}` の 3 クラス。各クラスから **代表値を 1 つ** 選ぶ。

#### 2.2 境界値の 5 種類を必ず洗い出す

各同値クラス間の境界について、以下を **すべて** リストする (1 つでも省略禁止):

| 境界カテゴリ | 取り得る値 |
|---|---|
| **最小値 (min)** | クラスの下端 |
| **最大値 (max)** | クラスの上端 |
| **min - ε / min + ε** | 境界の直前 / 直後 (整数なら ±1、浮動小数点なら最小単位、Date なら 1ms / 1day) |
| **max - ε / max + ε** | 同上 |
| **特異点** | `0` / 負値 / 空文字 / `null` / `undefined` / NaN / Infinity / 整数オーバーフロー / 浮動小数点誤差境界 / Date の DST 切替 / 月末 / 閏日 / Unicode 正規化境界 |

#### 2.3 期待結果の確定

各境界値について **期待結果を実行前に確定** する。期待結果は具体的な値 / 状態 / 例外型 で書く ("適切に処理される" は禁止)。

#### 2.4 Property-based testing による列挙漏れの補完（純粋ロジック）

> **前提ツール**: `fast-check` (+ `@fast-check/vitest`)。**backend / frontend とも導入済**（frontend は純粋 utils / composable 抽出純関数へ適用済・残る component 状態遷移／prop 境界等への展開は path-scope 漸進適用。導入状況の正は各 `package.json`）。**適用スコープ**: 入力→出力の純粋ロジック（utils / 集計 service 等）に絞る（property は純粋関数で最大価値を出し、metamorphic/fuzzing も内包するため）。

BVA は境界を **人手で列挙** するため、思いつかなかった境界は漏れる。入力 → 出力の純粋ロジックでは、BVA で具体境界を固めた上で **property-based test を併用** し、人手列挙の穴を機械生成で埋める:

- **不変条件 (property) を書く**: 個別の入出力ペアではなく「任意の入力で成り立つ性質」を宣言する。例: 往復 `parse(format(x)) === x` / 順序非依存 `sum(shuffle(xs)) === sum(xs)` / 冪等性 / 出力域の制約。
- **metamorphic relation もここで表現する**: 期待値の oracle が無い対象（複雑な集計等）は「入力を変換したときの出力の関係」を property として書く（専用ツールは使わず fast-check 上で表現）。
- 失敗時に得た **最小反例** は、BVA のテスト項目テーブルに固定ケースとして追加する（反例の永続化 + 回帰防止）。

### 3. 状態遷移分析 (STT)

state machine 型対象に対して以下を MECE に洗い出す:

#### 3.1 状態モデルの全列挙

- **状態 (State)** をすべて列挙 (例: `DRAFT` / `IN_PROGRESS` / `COMPLETED` / `REJECT`)
- **イベント (Event)** をすべて列挙 (例: `submit` / `approve` / `reject` / `cancel`)
- **初期状態** と **終了状態** を明示

#### 3.2 遷移表 (Transition Table) の作成

`(現在状態 × イベント) → 次状態` の表を **全マスを埋めて** 作成する:

|  | event_A | event_B | event_C |
|---|---|---|---|
| **state_1** | → state_2 | (無効) | → state_3 |
| **state_2** | (無効) | → state_3 | (無効) |
| **state_3** | (終了) | (終了) | (終了) |

#### 3.3 カバレッジ判定

- **0-switch カバレッジ (全遷移カバー)**: すべての有効遷移を最低 1 回テスト — **必須**
- **無効遷移カバー**: `(無効)` マスもテスト (例外 / 拒否を期待) — **必須**
- **1-switch カバレッジ (2 遷移シーケンス)**: 業務クリティカルなフローには適用
- **N-switch**: 過剰になりやすい、適用は判断

### 4. デシジョンテーブル (DT)

複数条件の組合せで挙動が分岐するロジックに対して以下を作成:

| 条件 (Cause) | R1 | R2 | R3 | R4 |
|---|---|---|---|---|
| 条件 A | T | T | F | F |
| 条件 B | T | F | T | F |
| **アクション (Effect)** |  |  |  |  |
| アクション X 実行 | ✅ | — | — | — |
| アクション Y 実行 | — | ✅ | ✅ | — |
| 例外発生 | — | — | — | ✅ |

- 全列 (= 全組合せ) を一度作る（条件 N 個なら 2^N 列）。ただし、矛盾 / 冗長な列は統合し、最終的なテーブルは統合後の列のみを記載する (don't care `-` で表記)
- 各列に対応するテストケースを 1 つ作る

### 5. ペアワイズ (組合せ爆発の管理)

パラメータが 3 以上 / 各 2 値以上で全網羅が 8 列以上になるとき:

- **2-wise カバレッジ**: 任意の 2 パラメータの組合せをすべてカバーする最小集合を選ぶ
- クリティカルパス (お金 / セキュリティ / データ整合性) は全網羅、それ以外はペアワイズ
- 工具: AllPairs / PICT 等の手計算 (件数少なければ手で導出)

### 6. エラー推測 (経験ベースのリスク追加)

以下のリスク領域は **必ず追加テスト** する (該当があれば):

| リスク | 典型例 |
|---|---|
| **off-by-one** | ループ境界 / 配列インデックス / 範囲 inclusive/exclusive |
| **race condition** | concurrent INSERT / SELECT → UPDATE シーケンス |
| **リソース枯渇** | 空集合 / null collection / 満タン / メモリ上限 |
| **時刻依存** | DST 切替日 / UTC ↔ JST 境界 / 月末 / 閏日 / うるう秒 / 過去日 / 未来日 |
| **i18n** | ひらがな⇄カタカナ / 大文字小文字 / Unicode NFC/NFD 正規化 / 全角半角 |
| **数値表現** | 整数オーバーフロー / 浮動小数点誤差 / `Number(string)` 変換失敗 / `NaN` / `Infinity` |
| **null/undefined 伝播** | optional chaining 不在 / `??` vs `||` 違い / `null` SQL semantics |
| **集合操作** | `DISTINCT` vs `GROUP BY` / 重複の意図 / 空集合の SUM |

### 7. テスト項目テーブルの作成

ステップ 2〜6 で洗い出した項目を **コード実装前に** 以下の表に整理する。表に書かれていない項目はテストしない (= 設計外):

| # | カテゴリ | 入力 / 状態 / 条件 | 期待結果 | 適用技法 |
|---|---|---|---|---|
| T-1 | 正常系 / 境界値 / 状態遷移 / 無効遷移 / 異常系 / リスク領域 | 具体的入力値 (例: `estimate=0`, `state=DRAFT, event=submit`) | 具体的出力 / 状態 / 例外型 (例: `returns 0`, `state=IN_PROGRESS`, `throws BadRequestError`) | BVA / STT / DT / ペアワイズ / エラー推測 |

テーブルは PRD 本文 / テスト設計コメント / TODO のいずれかに **永続化** する (口頭・チャットのみは禁止)。

### 8. テストケースの命名

`describe / it` 形式で、条件と期待結果を **両方** 名前に含める:

- 入力 → 出力: `it('estimate が 0 のとき total として 0 を返す')`
- 境界値: `it('最大長 -1 のとき作成成功')` / `it('最大長 + 1 のとき 400 を返す')`
- 状態遷移: `it('DRAFT 状態で submit すると IN_PROGRESS に遷移する')`
- 無効遷移: `it('COMPLETED 状態で submit は許可されず例外を投げる')`
- リスク領域: `it('DST 切替日を跨ぐ期間でも business days が正しい')`

## 禁止事項

- 「正常系」「異常系」「境界値」とだけ列挙して具体的入力値を確定しないこと
- 境界値の 5 種類 (min / max / min ± ε / max ± ε / 特異点) のうち 1 つでも省略すること
- 状態遷移を「主要遷移だけ」でカバーし、無効遷移をテストしないこと
- 期待結果を実行前に確定せず、実装後の値を見て後付けすること (`.claude/rules/verification.md` 違反)
- デシジョンテーブルが必要な複数条件分岐を、思いついた組合せ 2〜3 個でお茶を濁すこと
- ペアワイズ管理が必要なパラメータ群を全網羅 (= テスト爆発) すること、または逆に「2〜3 個適当に」で済ますこと
- 時刻 / i18n / 浮動小数点境界などのリスク領域を「実害ないだろう」と省略すること
- 集合演算 (`DISTINCT` vs `Σ` vs `重複加算`) の semantics 検証を 1 件しかカバーしないこと
- テスト項目テーブルを作らずにテストコードを書き始めること
- テスト名を `it('works')` `it('handles edge case')` などの抽象表現にすること
- backend の純粋ロジックの入力 → 出力で、property-based test (fast-check) を併用せず固定値の例だけで済ませること（§2.4）

## 検証

- テスト項目テーブルが PRD / テストファイルコメント / TODO のいずれかに永続化されている
- 境界値分析の 5 種類が全パラメータで洗い出されている
- 状態遷移型なら遷移表が全マス埋まっており、無効遷移にもテストがある
- 複数条件分岐ならデシジョンテーブルが作成され、各列にテストが対応する
- パラメータ組合せがペアワイズ管理されている (該当する場合)
- 各テスト項目に **具体的な** 入力値と期待結果が記載されている
- テスト名が `describe('<対象>') > it('<条件>のとき<期待結果>')` 構造で検証内容を明示している
- リスク領域チェックリスト (ステップ 6 の表) が走査され、該当項目がテストに追加されている
- backend の入力 → 出力の純粋ロジックで property-based test が併用され、発見した最小反例が固定ケース化されている
