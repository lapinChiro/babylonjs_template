# UX 設計手順

> テスト技法に対する `/test-design` と対称の位置づけ。`/test-design` が「テストを書く前にテスト技法を MECE に適用」するのと同じく、本 skill は「UI を作る前/検証する前に UX 手法を MECE に適用」する。

## トリガー

以下のいずれかに該当するとき。**2 つのモードを持つ**:

- **discovery モード**（何を作るか）: UI（frontend View / インタラクティブ component）を伴う PRD を作成するとき（`/prd-template` から）/ UI を伴う機能・修正に着手するとき（`/tdd` のテスト設計と並ぶ「前さばき」）。
- **review モード**（使えるか）: UI を伴う変更の完了処理時（`/tdd` ステップ0(ii) 完了処理列）/ frontend UI 変更の最終検証時。

backend のみ・UI 変更なしの場合はステップ 0 で T0 と判定して終了する。

## アクション

### ステップ 0: Tier 判定（両モード共通・最初に必ず実施）

変更規模を判定し、適用する手法の深さを決める（中庸＝機能規模で可変）。判定結果（Tier + 根拠）を PRD またはタスクに**明記**する。

| Tier | 対象 | discovery で行う | review で行う |
|---|---|---|---|
| **T0** | backend のみ / UI 変更なし | スキップ（「UI 変更なし=T0」と明記して終了） | スキップ |
| **T1** | 小さな UI 変更（既存画面への項目追加・文言変更等） | §1 ジョブストーリー（1 行） | §4 ヒューリスティック評価（縮約版＝関連原則のみ）+ §6 a11y 自動 |
| **T2** | 新規 View / 多段フロー / 新インタラクション | §1 + §2 ミニ・インパクトマップ + §3 仮説フロー/タスク分析 | §4 フル（10原則）+ §5 認知的ウォークスルー + §6 a11y（自動 + 手動） |
| **T3** | 大きな再定義トラック（複数画面横断） | §1〜§3 + 必要なら §7 ストーリーマッピング + プロトペルソナ更新 | T2 + ジャーニー全体検証（+ 任意で社内ゲリラテスト 2-3名） |

---

### discovery モード（何を作るか）

#### §1. ジョブストーリー

対象機能の中核ジョブを `When [状況], I want to [動機/力], so I can [期待成果]` で記述する。ペルソナ前提（"As a user"）を避け、**社内ロールの具体的な状況**を起点にする。例: `When レビューで複数の指摘が出て差し戻したとき, I want to 指摘ごとの対応状況を一覧で追いたい, so I can 全件解決してから再レビュー依頼できる`。

#### §2. ミニ・インパクトマップ（T2+）

`Why（この PRD のゴール）→ Who（社内ロール）→ How（その人の行動がどう変わるか）→ What（実装する機能）` を 1 段で書く。**What がゴール（Why）に寄与しない枝は削る**（スコープ正当化・YAGNI）。削った枝は PRD「対象外」へ。

#### §3. 仮説フロー / タスク分析（T2+・多段フロー）

ユーザーがゴールに到達するまでの**操作ステップ列**を分解する。各ステップで以下を洗い出し、受入基準・PRD スコープへ反映する:

- 分岐（成功 / 失敗 / 権限なし / データ 0 件）
- 必要な状態表現（loading / 空 / エラー / 完了）
- 摩擦点（入力が多い / 前画面に戻る必要 / 待ち時間）

#### §7. ストーリーマッピング（T3）

活動 → タスク → ストーリーの背骨（時系列・横軸）と詳細度（縦軸）を作り、MVP スライス（最初のリリース帯）を引く。スライスをそのまま PRD の作業ステップ / スコープへ落とす。

> discovery の全産物は PRD「背景・動機」「ゴール」「スコープ」「作業ステップ」へ**永続化**する（チャットのみは禁止）。

### review モード（使えるか）

#### §4. ヒューリスティック評価（Nielsen 10原則）

対象画面を**最低 2 周**点検し、違反を `場所 / 違反した原則番号 / 深刻度(0-4)` で列挙する（縮約版＝T1 は対象画面に関連する原則のみ）。10原則チェックリスト:

| # | 原則 | 点検観点 |
|---|---|---|
| ① | 状態の視認性 | loading / 保存中 / 成功 / 失敗 / 空 が見えるか |
| ② | 実世界との一致 | 用語が社内ロールの語彙か（専門用語の押し付けがないか） |
| ③ | ユーザーの制御と自由 | 取消 / やり直し / キャンセル / 戻るがあるか |
| ④ | 一貫性と標準 | 既存画面と同じパターン・同じ component か |
| ⑤ | エラー予防 | 破壊操作の確認 / 入力制約 / 無効値の事前ブロック |
| ⑥ | 想起より認識 | 判断に必要な情報が画面上にあるか（記憶を強いないか） |
| ⑦ | 柔軟性と効率 | 既定値 / ショートカット / 繰り返し操作の省力化 |
| ⑧ | 美的で最小限 | 情報過多・無関係要素がないか |
| ⑨ | エラーからの回復支援 | 原因と次の行動を示すメッセージか（コード羅列でないか） |
| ⑩ | ヘルプとドキュメント | 必要箇所に説明 / tooltip / マニュアル導線があるか |

#### §5. 認知的ウォークスルー（T2+）

discovery §3 で定義した主要タスクの**各ステップ**で、新規ユーザーの視点から次の 3 問に答える。No が出たステップが学習上の障壁:

1. ユーザーは「次に何をすべきか」が分かるか（正しい行動が見えるか）
2. ユーザーはその操作要素に気づけるか（発見可能か）
3. 操作後、ユーザーは「進んだ / 正しかった」と分かるか（フィードバックがあるか）

#### §6. アクセシビリティ評価（WCAG AA 抜粋）

a11y baseline を点検する（自動 = T1+、手動 = T2+）:

- **キーボードのみで全操作可能** / フォーカスリングが可視
- フォーム要素に `label`（または `aria-label`）/ 画像に `alt`
- **色のみに状態を依存させない**（テキスト/アイコン併用）
- エラーがテキストで識別できる / コントラスト比 AA

#### 指摘の処置

各指摘を「その場修正」か「`TODO` 起票」に振り分ける（`/refactoring-check` と同じ思想）。**深刻度 3-4 は原則その場修正**。処置を決めずに口頭で残さない。

### reference: プロトペルソナ（社内ロール・プロジェクト単位で 1 回作成し再利用）

discovery で「誰のため」を素早く想起するための暫定ペルソナ。社外顧客フェーズで実調査ベースのペルソナへ更新する前提:

| ロール | 主目標 | 主要画面 | 典型的な痛点 |
|---|---|---|---|
| QA レビュアー | レビューを滞りなく回し品質を担保 | ReviewDetail / ReviewerDashboard / ReviewStatistics | 指摘の追跡漏れ・差し戻し状況の把握 |
| PM / プロジェクト責任者 | 進捗・工数・リスクを俯瞰 | ProjectList / ProjectDetail / DailyMetrics | フェーズ進行・残工数/残日数の即時把握 |
| チームリーダー（開発者） | 自チームの作業状況を管理 | TeamDetail / UserTimeTrackings / FourKeys | 担当割り・実績入力の手間 |
| 管理者 | マスタ・権限を整備 | Permission* / 各 Management 系 | 設定の一貫性・操作ミス防止 |

## 禁止事項

- ステップ 0 の Tier 判定をスキップして手法を選ぶこと（深さの根拠が無くなる）。
- 該当 Tier・該当モードのステップを「主要なものだけ」で省略すること。
- discovery 産物を PRD に反映せず、チャット・記憶のみで進めること。
- review の指摘を「その場修正 / TODO 起票」のどちらにも振り分けず放置すること。
- floor（`ux-evaluation` rule）を満たしたことのみをもって T2 以上の UX 検証を完了とすること（floor は最低ライン、本 skill の review がフル評価）。
- ジョブストーリーをペルソナ前提の "As a user" に書き換えて状況・動機を省くこと。

## 検証

- Tier 判定（T0-T3）と根拠が PRD / タスクに明記されている。
- discovery モード: ジョブストーリー（+ Tier に応じたインパクトマップ / フロー分析 / ストーリーマップ）が PRD 本文に永続化されている。
- review モード: ヒューリスティック評価の指摘が `場所 / 原則 / 深刻度` で列挙され、各指摘が処置（その場修正 or TODO 起票）済み。
- T2+ で認知的ウォークスルーの 3 問が主要タスクの各ステップに適用されている。
- a11y baseline（キーボード / label / 色非依存 / エラー識別）が点検され、欠落が処置されている。
